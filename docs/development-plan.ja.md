# Mycel 開発予定

この文書は、今後検討している操作と機能の予定です。ここに書かれた内容は未実装の方針であり、実装時に操作名や細部を調整する可能性があります。

## 1. フォルダ直下のソート

### 目的

選択したフォルダの直下にあるファイルとフォルダを、名前や日時で並び替えられるようにします。マインドマップ上の手動配置を壊しすぎないよう、対象は選択フォルダの直下に限定します。

### 操作案

- フォルダを選択する
- 右クリックメニューから `並び替え` を開く
- 次のいずれかを選ぶ
  - `ファイル名 昇順`
  - `ファイル名 降順`
  - `更新日 昇順`
  - `更新日 降順`
  - `作成日 昇順`
  - `作成日 降順`
  - `手動順に戻す`

### 動作方針

- ソート対象は、選択フォルダの直下のファイルとフォルダだけにします。
- 下位フォルダの中身は、自動ではソートしません。
- ソート結果は `.mycel/order.json` に保存し、次回起動時も再現します。
- ソート後に手動ドラッグで並び替えた場合は、手動順を優先します。
- `手動順に戻す` を選ぶと、そのフォルダの保存済み並び順を解除し、通常の表示順へ戻します。
- 作成日を取得できない環境では、その項目を末尾に寄せます。

### 将来の拡張候補

- `このフォルダだけソート`
- `配下も含めてソート`
- フォルダ優先 / ファイル優先 / 混在ソートの切り替え

## 2. ノード検索

### 目的

大きなマップで目的のファイルやフォルダをすばやく見つけ、該当ノードへ移動できるようにします。

### 操作案

- `Ctrl + F` で検索バーを表示します。
- 入力中にファイル名とフォルダ名をリアルタイム検索します。
- `Enter` で次の検索結果へ移動します。
- `Shift + Enter` で前の検索結果へ移動します。
- `Esc` で検索を閉じます。

### 表示案

検索バーには検索語と件数を表示します。

```text
検索: [ report ]  3 / 12
```

一致したノードは、選択色とは別の淡い色で強調します。現在の検索位置にあるノードは選択状態にし、プレビューペインが表示されている場合は選択ファイルのプレビューを更新します。

### 折りたたみフォルダ内の検索

- 検索対象には、折りたたまれているフォルダ内のファイルとフォルダも含めます。
- 検索結果へ移動するときだけ、そのノードまでの親フォルダを表示します。
- 検索終了後は、検索前の折りたたみ状態へ戻すことを基本方針にします。
- 検索中にユーザーが手動で展開/折りたたみした場合は、その操作を優先します。

### 検索範囲

初期実装では、ファイル名とフォルダ名だけを対象にします。次の段階で、テキスト系ファイルの本文検索を追加します。

本文検索の対象候補:

- `.txt`
- `.md`
- `.html`
- `.csv`
- `.json`
- `.xml`
- `.cpp`
- `.h`
- `.py`

画像、動画、PDF、Office 文書などのバイナリファイルは本文検索の対象外にします。

## 3. `Key:` によるフィルター

### 目的

各テキストファイルに `Key:` 行を記述し、キーを指定してマップ表示を絞り込めるようにします。フォルダ構造だけでなく、文書の意味や分類に基づいてマインドマップを探索できるようにするための機能です。

### ファイル内の記述案

テキストファイル内に `Key:` で始まる行を置きます。`Key:` 行は既存方針どおりプレビューには表示しません。

```text
Subject: 顧客ヒアリング.md
Key: 顧客, ヒアリング, 重要

本文...
```

複数のキーは、カンマ区切りまたは空白区切りで指定できるようにします。大文字小文字は区別しない方針にします。

### 操作案

- `Ctrl + K` でキーフィルターバーを表示します。
- キーを入力すると、該当するファイルを含む経路だけを表示します。
- `Enter` で候補確定または次の該当ノードへ移動します。
- `Esc` でフィルターを解除します。
- フィルター中も通常の選択、プレビュー、編集は維持します。

### 表示案

```text
Key: [ 顧客 重要 ]  8 件
```

一致したファイルは強調表示します。該当ファイルへ到達するために必要な親フォルダは表示し、該当しない兄弟ノードは非表示または薄く表示する方式を検討します。

初期実装では、マップを読みやすくするために「該当しないノードを非表示」にする案を優先します。ただし、構造の把握を重視する場合は「薄く表示」に切り替えられる余地を残します。

### フィルター条件

初期実装では、入力したキーのいずれかに一致する OR 検索を基本にします。

例:

```text
顧客 重要
```

この場合、`顧客` または `重要` のどちらかを持つファイルを表示します。

将来の拡張として、すべてのキーを持つファイルだけを表示する AND 検索や、除外キーを指定する操作を検討します。

```text
顧客 +重要 -保留
```

### フォルダの扱い

- フォルダ自身に `Key:` は持たせず、配下に一致ファイルがある場合に表示します。
- 一致ファイルへ到達するための親フォルダは表示します。
- フィルター中にフォルダを折りたたんだ場合、そのフォルダ配下の一致項目は見えなくなります。

### キー一覧

ルート配下のテキストファイルを走査して、使用済みキーの一覧を作ります。キーフィルターバーでは、この一覧を候補として表示します。

候補表示の例:

- 入力中の文字に一致するキーを表示
- 最近使ったキーを上位に表示
- 件数が多いキーを上位に表示

初期実装では、入力補完よりも「使用済みキー一覧をリスト表示する」方式を優先します。

## 4. 検索とキーフィルターの組み合わせ

検索とキーフィルターは別機能として実装しますが、同時に使えるようにします。

例:

- `Key: 顧客` で顧客関連ファイルだけに絞る
- その状態で `Ctrl + F` により `議事録` を検索する
- 顧客関連ファイルの中から、名前または本文に `議事録` を含む項目へ移動する

### 優先順位

- キーフィルターは表示対象を絞り込みます。
- 検索は、現在表示対象になっているノードの中から移動先を探します。
- 検索結果がない場合は、フィルター外に一致があるかどうかを補助表示することを検討します。

## 5. 保存方針

### ソート

ソート結果は `.mycel/order.json` に保存します。

### 検索

検索語は基本的に保存しません。起動時に前回検索状態が残ると混乱しやすいためです。

将来の拡張として、最近使った検索語をアプリ設定に保存することは検討します。

### キーフィルター

初期実装では、フィルター状態は保存しません。次回起動時は通常表示に戻します。

将来の拡張として、名前付きフィルターを `.mycel/filters.json` に保存することを検討します。

## 6. 実装順序案

1. フォルダ直下のソート
2. ファイル名/フォルダ名検索
3. `Key:` の読み取りとキー一覧表示
4. `Key:` フィルター
5. 検索とキーフィルターの組み合わせ
6. テキスト本文検索
7. 名前付きフィルター保存

まずは、既存の選択、折りたたみ、プレビュー、編集、ファイル監視と衝突しにくい順に進めます。
